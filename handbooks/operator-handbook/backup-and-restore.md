# Backup and Restore

## What is backed up

`scripts/backup.sh` backs up the **PostgreSQL database only** — this holds essentially
all application state (roster, killmails, doctrines, encrypted OAuth tokens, audit log,
and so on). It does **not** back up:

- `.env` (secrets, including `TOKEN_ENCRYPTION_KEY`) — back this up separately (see
  below).
- The `eveimg_data` volume (mirrored EVE type images) — this is a rebuildable cache
  (`make import-assets` re-populates it), not a source of truth.
- TLS certificates in `./certs` — re-obtainable via `scripts/cert-init.sh` if lost.
- `static_data` / `nginx_cache` — both are regenerated by `collectstatic` and normal
  traffic respectively.

## Taking a backup

```bash
make backup
# equivalent to: scripts/backup.sh [output_dir]     (default output_dir: ./backups)
```

`scripts/backup.sh`:

1. Runs `pg_dump` **inside the `postgres` container** (no host PostgreSQL client
   needed), piping the output through `gzip`.
2. Writes to a temporary file first, so a dump that crashes partway never leaves a
   truncated file that looks valid.
3. **Verifies integrity**: `gzip -t` on the output, and a minimum size check (a failed
   dump is often only tens of bytes) — aborts with a non-zero exit if either check
   fails.
4. Atomically renames the verified temp file to
   `forca-<YYYYMMDD-HHMMSS>.sql.gz` in the output directory.
5. **Prunes old backups** beyond the retention count: `BACKUP_KEEP` (default **14**),
   keeping the newest `BACKUP_KEEP` dumps and deleting the rest.

The script never prints database credentials — it reads them from the container's own
environment.

```bash
BACKUP_KEEP=30 scripts/backup.sh /mnt/offsite-backups   # custom retention + destination
```

Because backups contain encrypted OAuth tokens and corporation data, store them with the
same care as the live database — see
[Data and Privacy § Backups](../data-and-privacy.md#backups).

## The nightly backup cron

The one-shot deploy script (`deploy/deploy-ubuntu-26.04.sh`) installs
`/etc/cron.daily/forca-db-backup`, which runs as the `forca` user with `BACKUP_KEEP=14`
against `/var/backups/forca`:

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "<app-src-dir>"
sudo -u forca BACKUP_KEEP=14 bash scripts/backup.sh "/var/backups/forca"
```

If you installed via the manual path (Path B in [Deployment](./deployment.md)), this
cron job is **not** installed automatically — set up your own recurring `make backup`
(or `scripts/backup.sh`) via cron or a systemd timer, and confirm it is writing files by
checking your chosen backup directory periodically (see the
[weekly checklist](./operations-runbook.md#weekly-checklist)).

Regardless of how backups are scheduled, copy them **off the host** — a backup that
lives only on the same disk as the database it protects does not protect against host
loss.

## Restoring from a backup

`scripts/restore.sh` is **destructive**: it drops and recreates the `public` schema
before loading the dump. It is guarded accordingly.

```bash
scripts/restore.sh ./backups/forca-20260707-030000.sql.gz
# or, non-interactively:
scripts/restore.sh ./backups/forca-20260707-030000.sql.gz --yes
```

Without `--yes`, it prompts:

```
This will OVERWRITE the current database with: <dump>
All data since that dump will be lost. Re-run with --yes to proceed.
Type the word RESTORE to continue:
```

What it does, in order:

1. Validates the dump file exists and is a valid gzip.
2. Requires explicit confirmation (the `RESTORE` prompt, or `--yes`).
3. **Takes a safety backup of the current database first** (via `scripts/backup.sh`) —
   so restoring is never a one-way trip away from your current state, even if you
   picked the wrong dump.
4. Inside the `postgres` container: `DROP SCHEMA public CASCADE; CREATE SCHEMA public;`
   then loads the dump with `psql -v ON_ERROR_STOP=1`.
5. Applies any migrations newer than the dump (`manage.py migrate --noinput`), so
   restoring an older backup onto a newer codebase still ends up schema-consistent.
6. Instructs you to run `make health` to verify.

## `TOKEN_ENCRYPTION_KEY` backup

`TOKEN_ENCRYPTION_KEY` is **not** included in a database backup — it lives only in
`.env`. Losing it makes every stored OAuth refresh token unrecoverable, meaning every
member must re-authorize with EVE SSO. This is a service disruption, not permanent data
loss, but it's disruptive enough to avoid deliberately:

- Back up `.env` (or at minimum `TOKEN_ENCRYPTION_KEY` and the other secrets it holds)
  to a secure location **separate from your database backups** — e.g. a password
  manager or secrets vault, not the same disk/bucket as `./backups`.
- If you ever rotate `TOKEN_ENCRYPTION_KEY` deliberately, treat it the same way: every
  member re-authorizes afterward.

## Disaster recovery considerations

- **Restoring onto a fresh host:** clone the repository, restore `.env` (with its
  original `TOKEN_ENCRYPTION_KEY` and database credentials matching the dump), deploy
  the stack (`make deploy`), then restore the database dump (`make restore
  FILE=...`) before running `make bootstrap` — the SDE/PI/image data restores from the
  same import commands, not from the database backup.
- **A restore always takes a fresh safety backup first** (see above), so a restore
  attempt gone wrong is itself recoverable.
- **`docker compose down -v` deletes every named volume**, including `pg_data` — this is
  intentionally destructive and is documented as such in
  [Troubleshooting § Starting over](./troubleshooting.md); never run it unless you mean
  to wipe the instance.
- Test your restore path periodically in a scratch environment (see the
  [monthly checklist](./operations-runbook.md#monthly-checklist)) — an untested backup
  is not a verified backup.
- Because `TOKEN_ENCRYPTION_KEY` and the database backup are two independent recovery
  artifacts, a full disaster-recovery plan needs **both** stored durably and separately.
