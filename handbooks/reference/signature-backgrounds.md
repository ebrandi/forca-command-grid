# Combat Signature Backgrounds

The Combat Signatures feature lets home-corp pilots render personalised banner images. Each
banner is composited over a **background** chosen from a fixed, curated library. This page is the
human-readable provenance record for that library; the machine-readable record is
`static/killboard/sigbg/manifest.json`.

## What the library is

Every background is **original abstract art generated procedurally** by the committed management
command `generate_signature_backgrounds` (in `apps/killboard/management/commands/`). There is no
image editor in the loop and, by deliberate design, **no upload path anywhere in the product** —
neither pilots nor administrators can add a raster image; administrators can only enable, disable,
categorise and order the built-in designs.

Each design is drawn from a **fixed integer seed** through Python's `random.Random(seed)` — never
the global RNG and never the clock — so the output is fully deterministic: regenerating the library
on the same Pillow/zlib build reproduces the committed bytes exactly. The generator draws each
design at twice the target size and downscales with LANCZOS for clean edges, then quantises to a
256-colour adaptive palette to keep every file small. The palette is anchored on the application's
own design tokens (see `frontend/tailwind.config.js`) so a banner reads as part of the app.

The committed assets live under `static/killboard/sigbg/<key>/` — one directory per design holding
`compact.png`, `standard.png`, `wide.png`, `card.png` and a `thumb.png` picker thumbnail — plus the
top-level `manifest.json`.

## Provenance and licensing

- **Origin.** All images are the original procedural output of the committed generator. They
  contain no photography, no hand-drawn art, and no third-party assets.
- **No CCP or third-party artwork.** The backgrounds contain **no CCP Games artwork, ship models,
  logos, insignia, screenshots or text**, and no artwork from any other third party. They are
  abstract compositions of light, colour and geometric primitives only.
- **EVE imagery is never public domain.** EVE-related imagery — character portraits, corporation
  and alliance logos, ship renders and type icons — is **never** treated as public domain or bundled
  into this library. Where the product shows such imagery elsewhere it is fetched, per request,
  from CCP's official Image Server exactly as the rest of the application already does, and subject
  to CCP's Developer Licence. It never appears in a signature background.
- **Licence.** Because the images are wholly generated by this repository's code, they are licensed
  under the **project's MIT licence**, like the rest of the source. The `manifest.json` records the
  same statement per design under `provenance` (`origin`, `source`, `license`, `notes`).

## Size presets and safe areas

Each design is committed at four fixed presets plus a thumbnail (no arbitrary dimensions):

| Preset | Dimensions | Typical use |
|---|---|---|
| `compact` | 468 × 120 | strict forum signatures |
| `standard` | 600 × 150 | general forum signatures |
| `wide` | 728 × 120 | wide forum/web banners |
| `card` | 600 × 200 | Discord and web embeds |
| `thumb` | 200 × 50 | background picker thumbnail |

Every design bakes in two **safe areas** that stay dark and low-detail so the renderer can lay text
over them at any preset: the **left 30%** (the portrait/name column) and a **central horizontal
band** (the primary stat line). The generator asserts this with `text_zone_ok`, which requires both
zones to fall below a mean-luminance and local-contrast threshold; the test suite enforces the same
gate on every committed file. Recommended overlay text colours are light on the dark zones — the
`ink`, `gold` and `cyan` tokens all pass comfortably.

## Regenerating and verifying

Run inside the application container so the Pillow and zlib versions match the committed bytes.

Regenerate the whole library (and `manifest.json`):

```
python manage.py generate_signature_backgrounds
```

Regenerate a single design in place (leaves the rest of the manifest untouched):

```
python manage.py generate_signature_backgrounds --only warp-tunnel
```

Verify that nothing has drifted — regenerates every design into a temporary directory, compares
sha256 checksums against the committed manifest and re-runs the safe-area gate, exiting non-zero on
any mismatch:

```
python manage.py generate_signature_backgrounds --check
```

Synchronise the database rows with the manifest (upsert by `key`; new designs are created enabled,
existing rows keep their admin enable/disable choice, and a key dropped from the manifest is
retired with `enabled=False` rather than deleted):

```
python manage.py sync_signature_backgrounds
```

The seed data migration `killboard/0027_seed_signature_backgrounds` runs the same sync on install,
so a fresh database starts with the full catalogue enabled.

## Adding a design

1. Add a render function and a registry entry to `generate_signature_backgrounds.py` with a **new,
   unused fixed seed** and the next `display_order`, in one of the six categories. Compose it from
   the existing primitives (starfields, nebulae, glows, warp streaks, tracers, grids, rings, cyno
   flares) and finish with the shared safe-area scrim — never darken the safe zones by hand.
2. Regenerate: `python manage.py generate_signature_backgrounds --only <key>` (or a full run). The
   command refuses to write any file whose safe areas fail `text_zone_ok`.
3. Confirm determinism and the committed checksums with `python manage.py
   generate_signature_backgrounds --check`.
4. Run the background tests (`tests/test_signature_backgrounds.py`) — they check dimensions,
   checksums, the contrast gate on every preset, and that the seed is unique.
5. Commit the new `static/killboard/sigbg/<key>/*.png` files and the updated `manifest.json`
   together with the generator change, and run `sync_signature_backgrounds` (or migrate) to surface
   the row. Keep this page's catalogue table in step with the manifest.
